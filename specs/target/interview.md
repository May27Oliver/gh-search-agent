這份文件是一份 AI Engineer 的帶回家評估作業（Take-Home Assessment），由 Oliver Chen 完成，日期為 2026 年 4 月 23 日。文件要求完成兩個部分的挑戰：

Part 1：Build a Tool, Break It, and Harden It（建立工具、破壞它、強化它）
目標是建立一個命令列介面（CLI）工具，能夠接受關於公開資料集的自然語言查詢，並將其轉換為結構化的 API 請求或資料庫查詢。
候選人需選擇一個公開的 API 或資料庫作為目標（例如 GitHub API、Wikidata 或 Data Commons），並完成以下四個要求：

Baseline Execution（基礎執行）：建構核心邏輯，工具必須能夠接受自然語言輸入、生成結構化查詢、實際執行查詢，並返回結果。
Break It（破壞測試）：測試工具的極限，輸入模糊指令、衝突的限制條件、拼寫錯誤以及非英語輸入，並記錄所發現的失敗案例。
Harden & Fix（強化修復）：修復最關鍵的失敗案例，讓工具在強化後必須能夠正確處理複雜的邊緣情況。
Explain（說明）：在 README 中提供詳細的技術說明，解釋為何某些剩餘的失敗案例在根本上難以解決。

Part 2：The Multi-Model Eval Challenge（多模型評估挑戰）
目標是設計一個評估管道（evaluation pipeline），測試不同的大型語言模型（LLM）在生成正確結構化查詢方面的表現，並完成以下五個要求：

Data Generation（資料生成）：程式化地收集或生成 30 個真實、多樣化的自然語言查詢，針對所選的領域，並包含邊緣案例、混亂的措辭和複雜的限制條件。
Ground Truth（標準答案）：為每個輸入撰寫完美、正確的結構化查詢作為標準答案，這些測試案例必須足夠複雜、具有對抗性，能夠主動讓模型失敗。
Execution & Iteration（執行與迭代）：使用至少 3 個不同的模型執行評估管道，這些模型必須包含開源和閉源模型的組合。
The Threshold（閾值要求）：持續迭代提示詞和管道，直到所有選定的模型在手動標記集上達到 >85% 的準確率。
Write-up（分析報告）：在 README 中提供涵蓋以下內容的分析：

Model Selection：如何選擇這 3+ 個模型，以及為何它們能夠達到準確率閾值？
Performance：比較各模型的表現，以及它們最初出錯的模式是什麼？
Learnings：從評估設計和建立結構化輸出的標準答案中學到了什麼？
